## 二、SPA iframe 架構（重要）

### 2-1 整體架構

```
dashboard.html（主框架）
├── topbar（固定，含 iframeActionBar）
├── sidebar（功能選單）
├── announce-bar（公告橫幅）
└── <iframe id="contentFrame">
      └── 各功能頁（hr.html / schedule.html / ...）
```

- 使用者點選側邊欄 → `goTo('xxx.html')` → 設定 iframe src → 載入功能頁
- **dashboard.html 的 topbar 永遠顯示**，功能頁的 topbar 在 iframe 內被隱藏
- `checkout.html` 例外：整頁跳轉（`window.location.href`），不在 iframe 內開

### 2-2 iframe 內 Topbar 隱藏機制

**原理**：iframe 載入後，dashboard 的 `frame.onload` 直接用同源 JS 對 iframe DOM 設 `data-embed="1"`，功能頁的 CSS selector `[data-embed="1"] .topbar` 就把 topbar 隱藏。

> ⚠️ **不能**用 URLSearchParams 判斷 embed，瀏覽器安全政策會 block iframe 的 URL 參數讀取。

**dashboard.html 的 `goTo()` 關鍵邏輯：**

```javascript
window.goTo = function(page) {
  const frame = document.getElementById('contentFrame');
  const sep = page.includes('?') ? '&' : '?';
  frame.src = page + sep + 'embed=1&v=4';   // v=4 繞過 CDN 快取
  frame.onload = function() {
    try {
      frame.contentDocument.documentElement.setAttribute('data-embed', '1');
      _syncIframeActions(frame);   // 把 iframe 功能按鈕搬到主框架 topbar
    } catch(_) {}
  };
  // ...sidebar active 狀態同步
};
```

**每個功能頁的 embed CSS（`[data-embed="1"]` selector）：**

```css
[data-embed="1"] .topbar,
[data-embed="1"] .topbar * {
  display: none !important;
  height: 0 !important;
  min-height: 0 !important;
  overflow: hidden !important;
}
```

> ⚠️ 新增功能頁**必須**加這段 CSS，否則 iframe 內 topbar 不會隱藏。

### 2-3 功能按鈕同步到主框架（iframeActionBar）

dashboard topbar 的 `登出` 按鈕前有 `<div id="iframeActionBar">`，每次 iframe 載入後，`_syncIframeActions()` 自動掃描 iframe `.topbar-right` 的功能按鈕，在主框架同步顯示代理按鈕。

**目前各頁面會同步的按鈕：**

| 頁面 | 同步到主框架的按鈕 | 備註 |
|---|---|---|
| oplog.html | ⬇ 匯出 CSV、↺ 重新整理 | |
| store_accounts.html | ＋ 新增帳號（⚙） | |
| task.html | ＋ 新增任務 | |
| backup.html | ＋ 立即備份 | |
| permissions.html | 儲存變更 | disabled 狀態同步 |

**排除規則**：`back-btn`、`role-chip`、日期時間 span（`todayLabel`、`clockLabel`）不搬移。

**切到 welcome 頁時**，iframeActionBar 自動清空。

### 2-4 iframe 內 back-btn 返回機制

功能頁的返回按鈕改用 `postMessage` 通知主框架導航，而非直接修改 `window.location`：

```javascript
onclick="if(window.self !== window.top){
  window.parent.postMessage({type:'navigate', page:'dashboard.html'}, '*');
} else {
  window.location.href = 'dashboard.html';
}"
```

dashboard 監聽此訊息並呼叫 `goTo(page)`。

## 四、權限系統（三層）

### 4-1 三層架構

| 層級 | 負責什麼 | 設定位置 |
|---|---|---|
| **role toggle** | 哪些角色可以進這個功能 | permissions.html → 功能權限分頁 |
| **locationType** | 只有央廚/只有門市/兩者都能進 | permissions.html → 店面類型分頁 |
| **locations** | 進去後看哪些店的資料 | auth-guard.js 回傳，功能頁直接用 |

### 4-2 auth-guard.js

- 集中式權限守衛，所有功能頁唯一入口
- 5 分鐘快取；permissions.html 儲存後清快取
- 版本參數：`import { authGuard, db, writeOpLog } from './auth-guard.js?v=2';`
- `locations` 已依 locationType 過濾完畢，功能頁直接用，**不需再自己過濾**
- Admin/Consultant 的 `userData.locations` 為空陣列；需全店清單時，讀 `hr/settings.workLocations`

```javascript
authGuard('moduleKey', ({ user, uid, role, locations, userData, db }) => {
  // locations：該用戶有權限的店面清單（已過濾）
  // role：admin / consultant / manager / employee
  // userData.name：顯示名稱
});
```

> ⚠️ 若 Firestore 沒有該 moduleKey 的設定，預設只有 admin 可進入。

## 五、新增功能頁完整流程

### 必做六步（缺一不可）

## 十、重要注意事項與常見陷阱

### auth-guard.js
- 修改 permissions.html 設定後，需**開新分頁**才生效（快取 TTL 5 分鐘）
- `hr.html` 保留次要 Firebase 實例用於建立帳號（非殘留，是設計）
- `locations` 對 admin/consultant 為空陣列，需全店清單時讀 `hr/settings.workLocations`
